# 技能

## 拉尔夫TUI产品需求文档生成器

### 名称：ralph - tui - prd

通过ralph - tui创建详细的、专为人工智能代理执行而优化的产品需求文档。

### 任务
从用户处获取功能描述
每次提出5 - 10个关键的澄清问题（带有字母选项）—— 一次一组
始终询问质量关卡（哪些命令必须通过）
每次得到回答后，如有必要，提出后续问题（自适应探索）
当你掌握足够的背景信息后，生成结构化的产品需求文档
将产品需求文档用 `[PRD]...[/PRD]` 标记包裹起来，以便TUI解析
重要提示：
不要开始实施，只需创建产品需求文档。

### 步骤1：澄清问题（迭代式）
一次提出一组问题。每个回答都应为你接下来的问题提供依据。重点关注：
- **问题/目标**：此功能解决了什么问题？（必填）
- **核心功能**：关键操作有哪些？（必填）
- **范围/边界**：它不应该做什么？用户做了系统应该给出什么具体的提示或响应？（必填）
- **成功标准**：我们如何知道它完成了？（必填）
- **集成**：它如何与现有功能相适配？（必填）
- **质量关卡**：每个用户故事必须通过哪些命令？（必填）

### 问题格式如下
1. 此功能的主要目标是什么？
    A. 改善用户入职体验
    B. 提高用户留存率
    C. 减轻支持负担
    D. 其他：[请说明]
2. 目标用户是谁？
    A. 仅新用户
    B. 仅现有用户
    C. 所有用户
    D. 仅管理员用户

这让用户可以通过回复 “1A, 2C” 进行快速迭代。

### 质量关卡问题（必填）
始终询问质量关卡 —— 这些是特定于项目的：
每个用户故事必须通过哪些质量命令？
    A. `pnpm typecheck && pnpm lint`
    B. `npm run typecheck && npm run lint`
    C. `bun run typecheck && bun run lint`
    D. 其他：[请指定你的命令]

对于用户界面相关的故事，是否应包括浏览器验证？
    A. 是的，使用 `dev - browser` 技能进行可视化验证
    B. 不，自动化测试就足够了

### 自适应提问
每次收到回复后，决定是：
- 提出后续问题（如果回答显示出复杂性）
- 询问新的方面（如果当前方面已明确）
- 生成产品需求文档（如果你掌握了足够的背景信息）
通常需要3 - 4轮提问。

### 步骤2：产品需求文档结构
生成包含以下部分的产品需求文档：
#### 1. 引言/概述
简要描述该功能以及它解决的问题。
#### 2. 目标
具体的、可衡量的目标（列表形式）。
#### 3. 质量关卡
关键内容：
列出每个用户故事必须通过的命令。

### 质量关卡
每个用户故事必须通过以下命令：
- `pnpm typecheck` —— 类型检查
- `pnpm lint` —— 代码检查

对于用户界面相关的故事，还应包括：
- 使用 `dev - browser` 技能在浏览器中进行验证

此部分将由转换工具（`ralph - tui - create - json`、`ralph - tui - create - beads`）提取，并添加到每个用户故事的验收标准中。
#### 4. 用户故事
每个用户故事需要：
- **标题**：简短的描述性名称
- **描述**：“作为 [用户]，我希望 [功能]，以便 [受益]”
- **验收标准**：可验证的 “完成” 定义清单
- **业务规则/约束**：明确边界和限制条件，输出明确直观的定义
- **边界值或输入域表**：如果存在需输入的字段值，需输出明确的字段是否必填，可输入字符是什么？可输入的字符长度是多少？
- **异常现象**：用户使用过程中操作的一些异常场景，系统需明确输出可量化的结果


每个用户故事应足够小，以便在一次人工智能代理会话中完成。
格式如下：
### US - 001: [标题]
**描述**：作为 [用户]，我希望 [功能]，以便 [受益]。
**验收标准**：
- [ ] 具体的可验证标准
- [ ] 另一个标准

注意：
不要在单个用户故事标准中包含质量关卡命令 —— 它们在 “质量关卡” 部分统一定义，并在转换过程中自动应用。
重要提示：
验收标准必须是可验证的，不能含糊不清。
“运行正常” 表述不佳。
“按钮在删除前显示确认对话框” 表述较好。
每个用户故事应能独立完成。

#### 5. 功能需求
具体功能的编号列表：
“FR - 1：系统必须允许用户……”
“FR - 2：当用户点击X时，系统必须……”
表述要明确清晰。
#### 6. 非目标（范围之外）
此功能不会包含的内容。这对管理项目范围至关重要。
#### 7. 技术考虑因素（可选）
已知的约束条件或依赖项
与现有系统的集成点
性能要求
#### 8. 成功指标
如何衡量成功？
#### 9. 未解决的问题
仍需解答的问题或需要澄清的方面。

### 为人工智能代理撰写内容
该产品需求文档将由人工智能编码代理通过ralph - tui执行。因此：
表述要明确清晰
用户故事应足够小（一次会话即可完成）
验收标准应尽可能可由机器验证
如果知道具体的文件路径，请包含在内
参考项目中现有的代码模式

### 输出格式
关键提示：
将最终的产品需求文档用标记包裹起来，以便TUI解析：
```
[PRD]
# PRD: 功能名称

## 概述
...

## 质量关卡
...

## 用户故事
...
[/PRD]
```

文件命名：
TUI将保存到 `./tasks/prd - [功能名称].md`

```
[PRD]
# 技能

## PRD: 拉尔夫TUI的暗黑模式

### 概述
为ralph - tui添加暗黑模式支持，以减轻长时间编排会话中的眼睛疲劳。该功能将应用于所有TUI组件，并将用户的偏好持久保存到配置文件中。

### 目标
- 在低光环境下提供舒适的观看体验
- 在不同会话中持久保存主题偏好
- 为所有TUI组件应用一致的样式

### 质量关卡
每个用户故事必须通过以下命令：
- `pnpm typecheck` —— 类型检查
- `pnpm lint` —— 代码检查
对于用户界面相关的故事，还应包括：
- 使用 `dev - browser` 技能在浏览器中进行验证

### 用户故事
#### US - 001: 添加主题配置
**描述**：作为用户，我希望设置我喜欢的主题（亮色/暗色），以便在不同会话中保持一致。
**验收标准**：
- [ ] 在 `.ralph - tui.yaml` 模式中添加 `theme` 字段
- [ ] 支持的值：“light”、“dark”、“system”
- [ ] 为了向后兼容，默认值为 “light”

#### US - 002: 创建暗黑主题调色板
**描述**：作为用户，我希望有一个柔和对比度的暗黑主题，对眼睛友好。
**验收标准**：
- [ ] 定义带有灰色调（非纯黑色）的暗黑调色板
- [ ] 确保足够的对比度（WCAG AA标准）
- [ ] 颜色适用于所有用户界面状态（选中、悬停、禁用）

#### US - 003: 将主题应用于TUI组件
**描述**：作为用户，我希望所有TUI组件都能遵循我的主题偏好。
**验收标准**：
- [ ] 标题组件使用主题颜色
- [ ] 任务列表使用主题颜色
- [ ] 详情面板使用主题颜色
- [ ] 进度条使用主题颜色
- [ ] 对话框使用主题颜色

#### US - 004: 在设置中添加主题切换按钮
**描述**：作为用户，我希望能在TUI设置中切换主题。
**验收标准**：
- [ ] 在设置视图中可见主题选项
- [ ] 更改立即生效，无需重启
- [ ] 更改持久保存到配置文件

### 功能需求
- FR - 1：主题设置必须可从 `.ralph - tui.yaml` 中读取
- FR - 2：主题必须在TUI启动时应用
- FR - 3：设置中的主题更改必须立即生效
- FR - 4：所有文本在两种主题下都必须保持可读性

### 非目标
- 系统主题自动检测（未来增强功能）
- 除亮色/暗色之外的自定义配色方案
- 按组件进行主题覆盖

### 技术考虑因素
- 如果可用，使用现有的OpenTUI主题功能
- 考虑为React组件创建一个主题上下文
- 主题更改时尽量减少重新渲染

### 成功指标
- 所有组件在暗黑模式下正确渲染
- 无无障碍对比度问题
- 主题在不同会话中持久保存

### 未解决的问题
在v2版本中，我们是否应该自动检测系统主题偏好？
[/PRD]
```

### 清单
在输出产品需求文档之前：
- [ ] 提出带有字母选项的澄清问题
- [ ] 询问质量关卡（必填）
- [ ] 在需要时提出后续问题
- [ ] 包含带有特定项目命令的质量关卡部分
- [ ] 用户故事规模小且可独立完成
- [ ] 用户故事不包含质量关卡命令（它们在质量关卡部分）
- [ ] 功能需求编号明确且表述清晰
- [ ] 非目标部分定义了明确的边界
- [ ] 产品需求文档用 `[PRD]...[/PRD]` 标记包裹起来
