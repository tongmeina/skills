# 测试报告编写指南

## 输入素材

输入数据来源于上游技能 **`skill/skills/zentao-bug-summary`** 输出的 **Markdown 文件**，通常已按 *创建者 → 状态 → 优先级* 结构化整理，并附标题列表。可能包含：

- **缺陷汇总.md**（必需，来自 `zentao-bug-summary`）：含总览统计、按状态/优先级分组的缺陷列表与标题
- **测试结果.md / 测试覆盖.md**（可选）：测试模块覆盖数据
- **潜在风险.md**（可选）：风险维度参考

> 与 `zentao-bug-summary` 的状态字段保持一致：**激活-待确认 / 激活-已确认（回归不通过） / 已解决 / 已延期 / 已关闭**。

## 分析范围（重要）

**本技能聚焦未解决缺陷的分析，不分析已解决缺陷。**

纳入分析的状态：
- **激活-待确认**
- **激活-已确认（回归不通过）**
- **已延期**

排除：**已解决**（仅在末尾统计数字时计入总量，不参与问题归纳与列表展示）。

## 输出结构

### 一、测试结果

**核心思路：仅基于「未解决缺陷」（激活 + 已延期）归纳问题类型，动态生成结论。**

**步骤1：解析缺陷汇总 Markdown**
- 从结构化分组中**筛选未解决缺陷**（激活-待确认、激活-已确认（回归不通过）、已延期）
- 统计未解决缺陷的优先级分布（一级/二级/三级/四级各 X 个）
- 从标题前缀（如「【账号管理】」「【消息管理】」）提取模块分布
- 提取问题关键词（如：翻译/i18n、时区、UI 拥挤、字体、白屏、报错、截断、参数错误等）

**步骤2：归纳问题类型**
- 仅对未解决缺陷的标题与描述归纳分类，常见方向如：国际化（翻译/时区/字体）、UI/排版、功能异常、易用性、数据准确性等
- **具体分类由实际未解决缺陷内容决定，不固定**
- **二级及以上**缺陷必须单独点名（高优问题）

**步骤3：生成测试结果**
采用如下格式：
> 本轮测试已完成 XX 模块的功能验证，当前仍有 **X 个未解决缺陷**，主要集中在 **【动态识别的分类1】 / 【动态识别的分类2】** 等方向，其中 **X 个高优问题（二级及以上）需优先修复**，X 个已延期问题建议下个版本跟进。
>
> - **【分类1】**：根据未解决缺陷归纳的具体描述
> - **【分类2】**：根据未解决缺陷归纳的具体描述

### 二、未解决问题汇总

仅列出**未解决缺陷**（激活-待确认 + 激活-已确认（回归不通过） + 已延期），按以下规则展示：

- **排序**：优先级从高到低（一级 → 二级 → 三级 → 四级）；同优先级内激活类排在已延期之前
- **格式**：每条一行，纯文本，便于复制，建议格式：
  `[优先级] [状态] 标题（模块前缀保留）`
- **数量**：超过 10 条则只展示前 10 条，并在底部说明被截断条数

底部统计（动态生成，固定格式）：
> 测试问题汇总：1 级：X 个；2 级：X 个；3 级：X 个；4 级：X 个，一共：X 个，剩余 X 个未解决，X 个待回归。

字段口径说明：
- **一共**：输入 Markdown 中的缺陷总量（含所有状态）
- **未解决**：激活-待确认 + 激活-已确认（回归不通过） + 已延期
- **待回归**：状态为 **已解决** 的缺陷数量（开发声明修复、待 QA 回归验证）
- **1–4 级**：对应输入中的 一级/二级/三级/四级（统计全量，不仅限未解决）

### 三、缺陷附件

在报告末尾附上「缺陷附件」小节，**以序号列出所有未关闭缺陷**（状态 ≠ 已关闭，即包含：激活-待确认 / 激活-已确认（回归不通过） / 已解决 / 已延期）。

输出规则：
- **按状态分组**，组内按优先级从高到低排序，组间顺序建议：激活-已确认（回归不通过） → 激活-待确认 → 已解决 → 已延期
- **序号全局连续递增**（1, 2, 3 …），不随分组重置
- 每条一行，纯文本，保留原始标题与模块前缀，建议格式：
  `序号. [优先级] 标题（模块前缀保留）`
- **不截断**：附件小节需完整列出所有未关闭缺陷，不受「未解决问题汇总」10 条上限约束

示例结构：
### 缺陷附件

激活-已确认（回归不通过）（X）

1. [二级] 【XXX】…
2. [三级] 【XXX】…

激活-待确认（X）

1. [二级] 【XXX】…

已解决（X）

1. [三级] 【XXX】…

…

已延期（X）

N. [四级] 【XXX】…

## 注意事项

- **仅基于输入 Markdown 中的「未解决缺陷」做结论**，所有结论必须来源于此，不得自行添加内容；已解决缺陷不参与问题归纳
- **避免使用表格**，全部文字描述，方便一键复制
- 量化精准，还原缺陷原貌；功能异常需说明**影响用户的具体场景**
- 领导关注重点：**结论前置 + 问题精炼**，二级及以上必须点名
- 若输入 Markdown 中未解决缺陷为 0，结论应明确写「本轮无遗留未解决缺陷」，并简述已解决情况
- 缺陷附件中的标题需与输入 Markdown 中保持一致，不重写、不改改词；若存在重复标题需保留原样
