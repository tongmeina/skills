# 技能

## 产品需求文档（PRD）

### 名称：prd

#### 概述
设计全面的、符合生产标准的产品需求文档（PRD），弥合业务愿景与技术执行之间的差距。此技能适用于现代软件系统，确保需求得到清晰定义。

# 技能

## 产品需求文档（PRD）

### 使用时机
在以下情况下使用此技能：
- 启动新产品或功能开发周期时
- 将模糊的想法转化为具体的技术规格时
- 定义人工智能驱动功能的需求时
- 利益相关者需要一个统一的项目范围“真相来源”时
- 用户要求“编写PRD”、“记录需求”或“规划功能”时

### 操作流程

#### 阶段1：发现（访谈）
在撰写PRD的第一行之前，你**必须**询问用户以填补知识空白。不要假设背景信息。
询问以下内容：
- **核心问题**：我们为什么现在要构建这个？
- **成功指标**：我们如何知道它起作用了？
- **约束条件**：预算、技术栈或截止日期如何？

#### 阶段2：分析与范围界定
综合用户的输入。识别依赖关系和隐藏的复杂性。
规划**用户流程**。
定义**非目标**以保护项目进度。

#### 阶段3：技术起草
使用以下严格的PRD架构生成文档。

### PRD质量标准

#### 需求质量
使用具体的、可衡量的标准。避免使用“快速”、“容易”或“直观”等表述。
# 模糊（差）
- 搜索应该快速并返回相关结果。
- 用户界面必须看起来现代且易于使用。

# 具体（好）
- 对于10k记录的数据集，搜索必须在200毫秒内返回结果。
- 搜索算法在基准评估中必须达到>=85%的Precision@10。
- 用户界面必须遵循“Vercel/Next.js”设计系统，并在灯塔可访问性评分中达到100%。

# 技能

## 产品需求文档（PRD）

### 严格的PRD架构
你**必须**按照以下确切结构输出：
#### 1. 执行摘要
- **问题陈述**：用1 - 2句话描述痛点。
- **建议解决方案**：用1 - 2句话描述解决方案。
- **成功标准**：3 - 5个可衡量的关键绩效指标。

#### 2. 用户体验与功能
- **用户角色**：这是为谁设计的？
- **用户故事**：作为[用户]，我想[行动]以便[受益]。
- **验收标准**：每个故事的“完成”定义的项目列表。
- **非目标**：我们不构建什么？

#### 3. 人工智能系统需求（如适用）
- **工具需求**：需要哪些工具和API？
- **评估策略**：如何衡量输出质量和准确性？

#### 4. 技术规格
- **架构概述**：数据流和组件交互。
- **集成点**：API、数据库和身份验证。
- **安全与隐私**：数据处理和合规性。

#### 5. 风险与路线图
- **分阶段推出**：最小可行产品（MVP） -> v1.1 -> v2.0。
- **技术风险**：延迟、成本或依赖项故障。

### 实施指南

#### 应该做（始终）
- **定义测试**：对于人工智能系统，指定如何测试和验证输出质量。
- **迭代**：提交草案并就特定部分征求反馈。

#### 不应该做（避免）
- **跳过发现阶段**：在没有先问至少两个澄清问题的情况下，绝不要撰写PRD。
- **臆想约束条件**：如果用户没有指定技术栈，询问或标记为待定（TBD）。

# 技能

## 产品需求文档（PRD）

### 示例：智能搜索系统

#### 1. 执行摘要
- **问题**：用户在海量存储库中难以找到特定的文档片段。
- **解决方案**：一个智能搜索系统，提供带有来源引用的直接答案。
- **成功标准**：
  - 减少50%的搜索时间。
  - 引用准确率 >= 95%。

#### 2. 用户故事
- **故事**：作为开发者，我想提出自然语言问题，这样我就不必猜测关键词。
- **验收标准**：
  - 支持多轮澄清。
  - 返回带有“复制”按钮的代码块。

#### 3. 人工智能系统架构
- **所需工具**：codesearch、grep、webfetch。

#### 4. 评估
- **基准测试**：用50个常见的开发者问题进行测试。
- **通过率**：90%必须与预期引用匹配。
