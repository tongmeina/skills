# 图片转测试PRD生成器

根据产品截图/原型图，生成一份结构严谨、测试友好的PRD文档，为测试工程师编写测试用例、前端与后端开发人员开发功能提供完整的需求依据。

## 适用场景

- 用户提供产品截图/原型图，需要生成PRD
- 用户说"帮我写PRD"、"生成需求文档"、"写测试PRD"
- 用户需要为测试团队准备需求文档
- 用户提到"字段规格"、"边界值"、"测试用例依据"

## 核心原则

1. **测试友好**：PRD内容必须足以支撑测试用例编写
2. **量化优先**：所有限制必须有具体数值，拒绝模糊描述
3. **场景完整**：覆盖正向流程、异常场景、边界条件
4. **可验证性**：每条需求都有明确的验收标准

```markdown
## 2. 用户故事

### US-001: [用户故事标题]

每个用户故事需要：
- **标题**：简短的描述性名称
- **描述**：“作为 [用户]，我希望 [功能]，以便 [受益]”
- **验收标准**：可验证的 “完成” 定义清单
- **业务规则/约束**：明确边界和限制条件，输出明确直观的定义
- **边界值或输入域表**：如果存在需输入的字段值，需输出明确的字段是否必填，可输入字符是什么？可输入的字符长度是多少？
- **异常现象**：用户使用过程中操作的一些异常场景，系统需明确输出可量化的结果


每个用户故事应足够小，以便在一次人工智能代理会话中完成。
格式如下：
### US - 001: [标题]
**描述**：作为 [用户]，我希望 [功能]，以便 [受益]。
**验收标准**：
- [ ] 具体的可验证标准
- [ ] 另一个标准

注意：
不要在单个用户故事标准中包含质量关卡命令 —— 它们在 “质量关卡” 部分统一定义，并在转换过程中自动应用。
重要提示：
验收标准必须是可验证的，不能含糊不清。
“运行正常” 表述不佳。
“按钮在删除前显示确认对话框” 表述较好。
每个用户故事应能独立完成。
```

## 图片分析流程

### Step 1: 识别界面元素

从图片中提取：
- 页面标题/模块名称
- 所有输入字段（输入框、下拉框、日期选择器等）
- 所有按钮（主按钮、次按钮、操作按钮）
- 列表/表格结构
- Tab/标签页
- 状态标识

### Step 2: 推断字段规格

对每个输入字段，推断：

| 推断项 | 方法 |
|--------|------|
| 数据类型 | 根据字段名称和用途推断 |
| 长度限制 | 参考行业标准（如手机号11位、身份证18位） |
| 格式要求 | 根据字段类型推断正则表达式 |
| 必填性 | 根据业务重要性推断 |
| 默认值 | 根据界面显示推断 |

**常用字段规格参考**：

```
卡号/编号: 6-20位数字, ^[0-9]{6,20}$
手机号: 11位数字, ^1[3-9]\d{9}$
邮箱: 最长100字符, ^[\w.-]+@[\w.-]+\.\w+$
用户名: 6-20位字母数字下划线, ^[a-zA-Z0-9_]{6,20}$
密码: 8-20位, 含大小写字母和数字
金额: 0.01-99999999.99, 保留2位小数
供应商名称: 1-50字符, 中英文数字
分组名称: 1-50字符, 中英文数字下划线
搜索框: 最长100字符
```

### Step 3: 分析交互逻辑

识别并记录：
- 按钮点击行为
- 表单验证规则
- 列表筛选/搜索逻辑
- 弹窗/对话框行为
- 状态切换逻辑

### Step 4: 推断业务规则

根据界面推断：
- 数据唯一性约束
- 关联关系（如设备ID与卡号一对一）
- 权限控制
- 状态流转规则

### Step 5: 列举异常场景

系统性列举：
- 输入验证失败场景
- 业务规则违反场景
- 并发/冲突场景
- 系统异常场景

```markdown

   - A. 仅[状态B]

   - D. 其他:____
```

## 质量检查清单

生成PRD后，检查以下项目：

### 字段规格检查
- [ ] 所有输入字段都有明确的长度/范围限制
- [ ] 所有输入字段都有数据类型定义
- [ ] 所有输入字段都标明是否必填
- [ ] 关键字段有正则表达式

### 边界值检查
- [ ] 数值类型字段有最小值/最大值
- [ ] 字符串类型字段有最小/最大长度
- [ ] 日期/时间类型字段有范围限制
- [ ] 列出了关键边界测试点

### 异常场景检查
- [ ] 覆盖所有输入验证失败场景
- [ ] 覆盖关键业务规则违反场景
- [ ] 覆盖常见系统异常场景
- [ ] 每个异常都有明确的提示信息

### 可测试性检查
- [ ] 每条需求都可验证
- [ ] 验收标准明确具体
- [ ] 状态流转规则清晰
- [ ] 权限控制明确

## 注意事项

1. **不猜测业务规则**：不确定的规则要向用户澄清
2. **使用具体数值**：避免"适当长度"、"合理范围"等模糊描述
3. **保持一致性**：同类字段使用相同的规格标准
4. **关注测试性**：每条需求都要考虑如何测试
5. **分层描述**：从概览到细节，逐步深入
