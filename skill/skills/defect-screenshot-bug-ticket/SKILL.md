# 缺陷截图 → Bug 单(精简量化版)

## 一、触发与输入

### 1.1 触发条件(任一)

1. 用户上传了缺陷相关截图(png / jpg / 聊天附件)

2. 用户说"帮我提 bug / 根据截图写 bug / 整理缺陷 / 识别这个问题"

### 1.2 必须输入

1. 缺陷截图(≥1 张)

### 1.3 可选输入

1. 一句简述:在哪个页面、做了什么、哪里不对

2. 模块 / 账号 / 环境 / 版本(用户提供则采用,否则写"待确认")

> 不主动要求用户补全字段;缺失项一律标注"待确认——原因:xxx",原因必须出自第 3.6 节枚举。

## 二、识别规则

### 2.1 标注优先级(最高,优先于全图扫描)

截图存在以下任一标注时,**缺陷主体必须落在标注区域**,禁止跑去描述无关区域:

1. 红色框选 / 红色箭头

2. 红色文字备注

3. 用户文字明确指向的字段 / 按钮 / 区域

无任何标注 → 才扫描全图。多个标注 → 按"红框 > 红箭头 > 红字 > 文字描述"优先,其余作为补充证据。

### 2.2 单图多缺陷拆分

1. 同图可独立成因的缺陷 ≥2 个 → 拆分多条 Bug 输出,标题分别为 `Bug-1 / Bug-2 / …`,每条仍走 8 块模板

2. 同一缺陷的多个表现(如同字段在多行重复异常) → 合并为一条,实际结果里枚举证据

### 2.3 非缺陷截图兜底

截图未发现可判定缺陷 → **不输出 8 块模板**,直接回复:

> 截图未识别到明显缺陷,请补充:1) 在哪个页面 2) 做了什么操作 3) 期望与实际的差异

### 2.4 动态行为类缺陷(无 loading / 卡顿 / 重复请求 / 闪退 / 时序问题)

静态截图无法判定动态行为。处理口径:

1. 用户已提供文字描述或录屏 → 正常输出 8 块模板

2. 仅有静态截图 → 严重程度 / 优先级 / 操作步骤标"待确认——原因:动态行为单图不可判定",并提示用户补录屏或文字时序

### 2.5 模块名识别(用于 Bug 标题)

1. 模块名 = **截图中当前被选中(高亮 / 激活态)的导航项文案**,优先级:

   1. 左侧菜单高亮项

   2. 顶部 Tab 高亮项

   3. 面包屑末级文案

   4. 页面主标题

2. Bug 标题固定格式:`【<模块名>】<现象+关键值>`

3. 一级 + 二级菜单同时高亮 → **取最末级的高亮项**(最具体那个)

4. 截图无任何高亮导航项 → 写 `【模块待确认】`,无需在前置条件中补充入口路径

5. 不允许根据表格列名 / 按钮文案 / 弹窗标题反推模块名

## 三、判定口径

### 3.1 严重程度(按影响面)

1. **S1**:崩溃 / 数据丢失 / 核心链路(如 SOS 求救)完全不可用

2. **S2**:主流程阻塞或关键功能不可用,但可绕过

3. **S3**:功能异常但不阻塞主流程

4. **S4**:UI / 文案 / 展示瑕疵

### 3.2 优先级(按修复紧迫性)

1. **P1**:必须立即修复(影响上线 / 大量用户 / 核心链路)

2. **P2**:应尽快修复(影响明显但可临时规避)

3. **P3**:可排期修复(体验类 / 低频 / 影响小)

### 3.3 严重 ↔ 优先级 默认映射

| 严重程度 | 默认优先级 |
| --- | --- |
| S1 | P1 |
| S2 | P1 / P2 |
| S3 | P2 / P3 |
| S4 | P3 |

> 默认建议,可被截图证据或用户描述覆写;**严禁出现 S1 配 P3、S4 配 P1 等明显不匹配组合**。

### 3.4 缺陷类型(枚举)

功能缺陷 / UI缺陷 / 数据一致性 / 兼容性 / 性能 / 权限 / 文案国际化 / 接口异常

### 3.5 缺陷类型 → 量化规则库 默认映射

| 缺陷类型 | 默认引用规则库 |
| --- | --- |
| 数据一致性 | B 数据/格式 + E 一致性/刷新 |
| 文案国际化 | A 文案/国际化 |
| UI缺陷 | A + B |
| 接口异常 | C 交互/状态 + D 校验/错误提示 |
| 性能 | C |
| 功能缺陷 | C + D |
| 权限 | D |
| 兼容性 | A + B(分辨率 / 格式相关) |

> 写预期结果时,**先按映射定位规则库分组,再从分组中挑符合截图证据的条目**,避免跨分组乱选。

### 3.6 "待确认——原因:xxx" 固定枚举

仅允许以下原因,禁止自由发挥:

1. 截图未显示

2. 截图模糊

3. 用户未提供

4. 动态行为单图不可判定

5. 与现有 PRD 冲突

## 四、预期结果规则

1. **直接写结论**,不写 `【优化点】` 前缀,不分段写"规则 + 量化"

2. 每条**一行写完**,必须自带量化标准(数值 / 枚举 / 格式 / 上限 / 耗时)

3. **最多 3 条**,排序优先级:数据正确性 > 交互一致性 > 文案UI

4. **禁止含糊词**:应正常 / 应优化 / 应合理 / 体验更好 / 尽快 / 友好

5. **字段名 / 期望值 / 格式样例统一加反引号**(如 `Order list`、`YYYY-MM-DD HH:mm:ss`、`0/10条`),方便研发搜索

### 4.1 内置量化规则库(无 PRD 时的默认口径)

**A. 文案 / 国际化**

1. 文案必须逐字一致(含标点、大小写、空格)

2. 多语言缺失语言包不得显示原始 key(如 `xxx.title`)

**B. 数据 / 格式**

1. 时间格式 `YYYY-MM-DD HH:mm:ss`

2. 经纬度保留 6 位小数

3. 金额保留 2 位小数

4. 列表分页默认 20 条/页

5. 计数 / 额度类字段值不允许为负数,展示范围明确

**C. 交互 / 状态**

1. 点击触发接口 ≤1s 出现 loading / 禁用态

2. 请求完成 ≤3s 展示结果;超时必须给出明确原因

3. 请求进行中按钮置灰,连续点击 N 次抓包请求数 = 1

**D. 校验 / 错误提示**

1. 必填项空值必须拦截,提示精确到字段

2. 接口失败提示必须包含原因或错误码,禁止仅写"失败"

**E. 一致性 / 刷新**

1. 新增 / 删除 / 修改后,列表与详情 ≤3s 内一致

2. 需手动刷新时必须提供明确入口(下拉刷新 / 刷新按钮)

## 五、输出规则(强约束)

1. **只输出下面 8 个块**,顺序固定,不增不减:
   Bug标题 / 严重程度 / 优先级 / 缺陷类型 / 前置条件 / 操作步骤 / 实际结果 / 预期结果

2. **列表统一有序列表**,编号 `1. 2. 3. …` 递增;**每个列表项之间必须空一行**(loose list),确保自动换行渲染

3. **多缺陷**:拆为多条 Bug,各自独立 8 块,标题前缀 `Bug-1 / Bug-2 / …`

4. **多图**:同一缺陷的多张图合并为同一条 Bug,步骤里用"见截图1 / 截图2"定位

5. **敏感信息**:手机号 / 账号等照抄截图;用户要求脱敏再改为「已脱敏」

6. **多语言**:模板章节标题固定中文(禅道工单语言),引述截图原文照抄(中 / 英 / 混合)

### 5.1 Bug 标题约束

1. **≤32 字**,超出必须裁剪

2. **禁词**:问题、异常、bug、不对、报错、有点、似乎、好像

3. 必须包含"**现象 + 关键值 / 字段**"(如 `短信额度展示为负数「-1/10条」`),纯描述性词汇不达标

4. 格式固定:`【<模块名>】<现象+关键值>`

### 5.2 操作步骤约束

1. **1~6 条**,覆盖到能定位问题为止,不强行凑数也不省略关键操作

2. 每条以动词开头(进入 / 点击 / 输入 / 切换 / 上传 / 查看 / 观察)

### 5.3 实际结果约束

1. 第 1 条必填:**照抄截图错误现象**(文案 / 数值 / 状态),关键值加反引号

2. 第 2 条「影响」**仅 S1 / S2 必填**;S3 / S4 可省略此条

## 六、统一输出模板

```text
## Bug标题
【<截图中高亮导航项>】<现象+关键值,≤32字>

## 严重程度
S1 / S2 / S3 / S4(无法判断写:待确认——原因:<枚举>)

## 优先级
P1 / P2 / P3(无法判断写:待确认——原因:<枚举>)

## 缺陷类型
功能缺陷 / UI缺陷 / 数据一致性 / 兼容性 / 性能 / 权限 / 文案国际化 / 接口异常

## 前置条件
1. <必要数据准备:一句话描述列表/页面/记录的存在条件>

## 操作步骤
1. 进入 xxx 页面(见截图1)

2. 执行 xxx 操作(点击 / 输入 / 切换 / 上传)

3. 观察 xxx 区域 / 字段 / 状态

## 实际结果
1. 截图显示:`xxx`(照抄关键值)

2. 影响:xxx(仅 S1/S2 必填,S3/S4 省略此条)

## 预期结果
1. <一句话量化结论,加反引号>

2. <一句话量化结论,加反引号>
```

## 七、参考示例(基于"我的套餐"截图)

```text
## Bug标题
【我的套餐】短信额度展示为负数「-1/10条」

## 严重程度
S3

## 优先级
P2

## 缺陷类型
数据一致性

## 前置条件
1. 我的套餐列表存在至少 1 条套餐记录,且列表展示「短信额度」字段

## 操作步骤
1. 进入「我的套餐」页面(见截图1)

2. 设备类型选择「天通救援棒」,状态选择「全部」

3. 查看设备卡号 5052 的「短信额度」列

## 实际结果
1. 设备 5052 的「短信额度」展示为 `-1/10条`,出现负数

2. 同列其他设备均为非负值(范围 `0/10条 ~ 10/10条`)

## 预期结果
1. 短信额度展示格式固定为 `<已用>/<总额度>条`,已用值 ∈ [0, 总额度],不允许为负数

2. 套餐状态为「已过期」时,短信额度统一展示为 `0/<总额度>条`
```

## 八、不做的事

1. 不主动加载全量 PRD / 用例(等用户 @ 引用)

2. 不要求用户安装工具 / 跑脚本

3. 不扩展成测试方案、根因分析、长篇优化建议

4. 不输出模板外的章节(禅道字段大表、自检清单、提取说明、缺陷分析表格等全部删除)

5. 不编造截图未出现的字段;缺失一律写"待确认——原因:<3.6 枚举>"

6. 不根据表格列名 / 按钮文案反推模块名
