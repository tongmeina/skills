# 缺陷截图 → Bug 单

## 一、何时触发

以下任一条件满足即触发本技能：
1、用户上传了缺陷相关截图（png / jpg / 聊天附件）
2、用户要求"整理 Bug""识别 Bug 单""写缺陷描述""写缺陷分析"
3、用户说"帮我提 Bug""这个问题帮我写一下"

## 二、输入规范（引导用户提供）

### 2.1 必须输入
1、**缺陷截图**（至少一张）

### 2.2 建议输入（结构化补充）

如果用户提供了以下信息，优先采用用户描述的内容：

```
【模块】xxx（如：我的信息 → 我的套餐）
【环境】测试环境 / 正式环境
【账号】xxx
【版本】v1.x.x
【简述】做了什么操作，出了什么问题
```

如果用户未提供，从截图中推断并标注"根据截图推断"。

### 2.3 可选输入（按需 @ 引用，节省 token）

1、`@docs/prd/模块名.md` — 相关模块的 PRD 摘要
2、`@docs/testcases/TC-模块名.md` — 相关模块的测试用例
3、`@Codebase` — 语义搜索项目文档

> **重要**：不要主动加载全量 PRD 或用例文档。仅在用户 @ 引用时读取，
> 或在用户明确要求"结合需求/用例"时，提示用户 @ 对应模块文件。

## 三、识别两类版式

| 版式 | 图中常见结构 | 输出章节 |
| --- | --- | --- |
| **A. 缺陷描述** | Bug 标题、前置条件、复现步骤、实际结果、预期结果 | 模板 A |
| **B. 缺陷分析** | 缺陷分析表格、优化建议列表、严重等级 | 模板 B |

一张图可能只有 A、只有 B，或两者都有：有的则输出，没有的整块省略。

## 四、输出格式要求

1、**语言**：与截图正文一致（一般为简体中文）。
2、**列表**：全文**仅允许有序列表**，且同一列表内必须按 `1、2、3…` **递增编号**（含前置条件 / 复现步骤 / 实际结果 / 预期结果 / 优化建议 / 自检清单等），禁止使用无序列表 `- `。
3、**标题层级**：使用模板中的 `##` / `###`。
4、**模块名**：若截图标题含 `【模块】`，原样保留；若无且可推断，补 `【模块】` 并标注"模块名为根据截图推断"。
5、**敏感信息**：手机号、账号等照抄截图；若用户要求脱敏，再改为「已脱敏」。
6、**多图场景**：按上传顺序编号（截图1、截图2…），合并到同一条 Bug 单，复现步骤里标注"见截图X"。
7、**不要编造**：截图中看不清或未出现的信息，用简短说明标出（如「截图模糊：xxx 无法辨认」）。
8、**可衡量/可验收**：预期结果与优化方案必须可验证，优先给出**明确文案/字段值/单位/数量/阈值**，避免仅写"应正常/应优化/应展示"。

可衡量写法示例（按实际截图内容替换）：
1、文案/翻译：将「订单列表」英文文案修正为 **`Order list`**（大小写按设计稿/国际化规范）。
2、数量/上限：列表默认展示 **20** 条，支持上拉加载至 **100** 条（或按 PRD/现网一致性）。
3、时间/性能：点击「发送」后 **≤ 1s** 出现发送中态，**≤ 3s** 返回发送结果（超时提示明确原因）。
4、格式/单位：位置展示为 **经纬度保留 6 位小数**，时间为 **`YYYY-MM-DD HH:mm:ss`**。


## 五、输出模板

### 模板 A：Bug 标题 + 缺陷描述 + 禅道字段

```
## Bug 标题

【模块名】一句话概括问题（与截图一致）

## 禅道字段

1、**所属产品**：星联应急保障平台 / 星联应急叫应小程序
2、**所属模块**：（从截图/描述获取）
3、**影响版本**：（从描述获取，未提供则标"待确认"）
4、**测试环境**：（从描述获取）
5、**严重程度**：S1 / S2 / S3 / S4
6、**优先级**：P1 / P2 / P3
7、**缺陷类型**：功能缺陷 / UI缺陷 / 数据一致性 / 兼容性 / 性能
8、**指派给**：（待指派）

## 缺陷描述

### 前置条件

1、账号/角色：（如：普通用户 / 管理员；未提供则标“待确认/根据截图推断”）
2、入口路径：（如：我的信息 → 我的订单；不确定则标“待确认”）
3、数据准备：（如：存在至少 1 条订单；或设备已绑定；按截图补充）

### 复现步骤

1、（入口）进入 xxx 页面（见截图X）
2、（操作）点击/输入/切换 xxx
3、（触发）观察 xxx 区域/字段/按钮状态变化

### 实际结果
1、（现象）xxx 字段显示为 xxx / 文案为 xxx / 状态为 xxx（与截图一致）
2、（影响）导致用户无法 xxx / 误解 xxx（若能从截图推断，否则省略）

### 预期结果
1、（应显示的明确值）xxx 字段应为 **`<期望值>`**（如：将「订单列表」英文文案修正为 **`Order list`**）
2、（可验收标准）展示/校验/交互应满足 **<阈值/规则>**（如：列表默认 20 条；点击后 1s 内出现加载态）

```

**严重程度判断参考**：
1、**S1**：系统崩溃、数据丢失、核心链路（SOS 求救）完全不可用
2、**S2**：主流程阻塞但有临时替代方案
3、**S3**：功能异常但不影响核心业务
4、**S4**：UI 体验优化、文案错误等

### 模板 B：缺陷分析 + 优化建议 + 严重等级

```
## 缺陷分析

| 缺陷维度 | 问题描述 | 优化方案 |
| --- | --- | --- |
| 文案/国际化 | 「订单列表」未翻译为英文/翻译不一致 | 统一为 **`Order list`**（与设计稿/语言包 key 保持一致），并补齐缺失语言包项 |
| 交互/状态 | 点击按钮无 loading/重复点击触发多次请求 | 增加 **loading** 与 **防重复点击**（同一请求期间按钮置灰；重复点击不再发起） |
| 容错/提示 | 接口失败仅 toast“失败”无原因 | 按错误码展示明确提示（如：超时/无网络/权限不足），并提供重试入口 |

## 优化建议

1、优先补齐**可验收**的期望值：明确文案/字段值/数量/阈值（例：`Order list`、默认 20 条、点击 1s 内出现 loading）
2、若涉及接口：补充「请求参数/返回字段/错误码」与前端展示映射，确保研发可直接定位

## 严重等级

**等级**：Px（或截图中的写法）

（截图中严重等级下的说明段落，完整复述。）

```

## 六、交付顺序

1、同时存在 A + B → 先输出 **模板 A**，空一行，再输出 **模板 B**。
2、图中只有分析页 → 只输出模板 B，开头说明"本截图为缺陷分析版式"。
3、末尾附 **自检清单** 和 **提取说明**。


## 七、提取说明（每次输出末尾附上）

```
## 提取说明

1、**置信度**：高 / 中 / 低
2、**模糊字段**：有 / 无（如有，列出具体字段）
3、**建议核对**：xxx

```

## 八、不必做的事

1、不主动加载全量 PRD / 用例（等用户 @ 引用）
2、不要求用户跑脚本或安装工具
3、不主动扩展成测试方案或长篇根因分析（除非用户另提要求）
4、不编造截图中不存在的信息
```

```
