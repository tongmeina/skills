# Bug 报告输出 + 禅道自动写入

本技能实现两步工作流：**先输出结构化 Bug 报告 → 再自动写入禅道**。

## 第一步：输出结构化 Bug 报告

收到用户描述的问题后，按以下模板输出完整的 Bug 报告。输出前必须确认用户已提供足够的信息（项目名称、问题描述），若信息不足则主动询问。

### Bug 标准格式模板

```markdown
## 【{模块名}】{简短标题}

**前置条件**

1、{条件1}
2、{条件2}
...

**复现步骤**

1、{步骤1}
2、{步骤2}
...

**实际结果**

1、{实际现象1}，例如：{具体截图或日志示例}
2、{实际现象2}

**预期结果**

1、{期望行为1}
2、{期望行为2}
```

### 输出规范

| 字段 | 规则 |
|------|------|
| **模块名** | 用方括号标注所属功能模块，如【求救群聊】【国际化-控制台】【套餐商城】 |
| **标题** | 15~30 字，包含模块 + 问题本质，如「会话内发送失败后错误提示中英混排」 |
| **前置条件** | 列出复现所需的前置状态，如语言环境、页面位置、账号权限等 |
| **复现步骤** | 每步一行，编号排列，操作路径清晰到可直接执行 |
| **实际结果** | 客观描述看到的现象，附关键截图引用或日志原文 |
| **预期结果** | 描述正确行为应是什么样的 |
| **序号（重要）** | 凡需要**有序排列**的条目，**一律**使用 **`1、` `2、` `3、`** 依次类推（阿拉伯数字 + 全角顿号「、」）。**禁止**混用 `1.` `1)` `（1）` `-` 等其它序号样式；生成 `steps-file` 时也必须遵守（脚本仅将 `^\d+、` 转为有序列表）。 |

### 缺陷等级参考（根据影响范围自动判定）

| 等级 | 含义 | 典型场景 |
|------|------|----------|
| 二级 (severity=2) | 核心功能不可用、数据丢失风险 | 支付流程崩溃、消息发不出、登录失败 |
| 三级 (severity=3) | 功能异常但不阻塞主流程 | UI错乱、提示文案错误、部分字段未翻译 |
| 四级 (severity=4) | 轻微体验问题、建议优化 | 排版微调、文案不够优雅 |

### 优先级参考（pri）

| 值 | 含义 | 使用场景 |
|----|------|----------|
| 2 | 高 | 影响大量用户或核心链路 |
| 3 | 中 | 正常缺陷，常规处理 |
| 4 | 低 | 非紧急，可延后修复 |

## 第二步：自动写入禅道

用户确认 Bug 内容无误后，调用底层脚本将 Bug 写入禅道。

### 写入脚本

使用已有的 `zentao-bug-create.mjs`：

```
skills/skill/mcp/scripts/zentao-bug-create.mjs
```

### 构建命令

将第一步输出的 Bug 报告保存为临时文件，然后构造命令：

```bash
node "skills/skill/mcp/scripts/zentao-bug-create.mjs" ^
  --project-name "{项目名称}" ^
  --title "{Bug 标题}" ^
  --steps-file "{临时文件路径}" ^
  --severity {严重程度} ^
  --pri {优先级} ^
  --opened-build {版本号} ^
  --execution {迭代/测试模块ID} ^
  --attach "{截图路径}"
```

`--attach` 可重复；路径为本地文件（png/jpg 等）。

### 产品 ID 解析顺序（方案一，与缺陷汇总同源）

写入脚本在**只提供 `--project-name`**（或仅 `--project-id` 且无产品参数）时，按顺序自动得到「产品 ID」，**无需用户手填**，除非以下各步均失败：

1. **项目详情**：`GET /api.php/v1/projects/{项目ID}`，从返回中的 `product` / `products` / `linkedProducts` 等字段解析产品 ID（适配不同禅道版本）。
2. **项目下已有缺陷（方案一）**：与 **`zentao-bug-summary`** / `zentao-bugs-summary.mjs` 相同，调用 **`GET /api.php/v1/projects/{项目ID}/bugs`**（分页取前若干条），从缺陷对象上的 **`product`**（或 `productID` / `productId`）推断产品 ID——即「你在界面里进项目→测试→历史缺陷」所对应的数据源。
3. **仍失败时**：请用户指定 **`--product-id`**（禅道 → 产品 → 列表/详情中的数字 ID），或先在该项目下创建/保留至少一条缺陷后再重试。

创建请求仍为 **`POST /api.php/v1/products/{产品ID}/bugs`**，并在请求体中写入 **`project`**（项目 ID），使新缺陷与你在禅道里选的**项目**一致。

### 写入禅道时的正文结构（与第一步对齐）

第一步报告含 **前置条件 / 复现步骤 / 实际结果 / 预期结果** 四块时，写入 `steps-file` **须四块齐全**，勿为迁就禅道默认的 `[步骤][结果][期望]` 而省略 **前置条件**（此前若只写了三段模板，属于**流程约定**未写进技能，易造成「前置条件被吃掉」）。推荐在 `steps-file` 中显式标出小节，例如：

```markdown
[前置条件]
1、……
2、……

[步骤]
1、……

[结果]
1、……
2、……

[期望]
1、……
```

**实际结果、预期结果**若有多条，请每条单独一行且使用 **`1、` `2、`** 编号（全角顿号、与第一步模板一致）。脚本会将连续编号行转为 HTML **有序列表**，禅道里会显示为有序「多条实际结果」，而不会挤成一段。

### 序号规范（写入 steps-file 时强制对齐）

1、**格式**：仅使用 **`{n}、`**（如 `1、` `2、` `3、`），每条占一行；每个小节（`[前置条件]`、`[步骤]`、`[结果]`、`[期望]`）内**各自从 `1、` 重新编号**，依次递增。  
2、**整理**：若用户描述或草稿里出现了 `1.` / `（1）` / 多级编号等，**在写入 steps-file 前**应改写成上述 `1、` `2、` 形式，保持全篇一致。  
3、**非序号段落**：小节标题（如 `[结果]`）、无编号的说明句，单独成行，不要加 `1、`。

### 写入禅道的 steps-file 完整示例

以下为符合序号规范、可直接作为 `--steps-file` 保存的示例（字段内容可按实际缺陷替换）：

```markdown
[前置条件]
1、界面语言为 English。
2、已进入 Message → Communication Logs，并可打开与终端的会话。

[步骤]
1、在 Communication Logs 中打开设备会话（示例：钢铁侠(66006611)）。
2、触发向终端发送失败且返回余额/充值类错误的场景。
3、查看会话区内红色失败提示全文。

[结果]
1、提示为 Failed to send to terminal(客服余额不足，请充值后重试)，括号内仍为中文。
2、与同页英文 UI 不一致。

[期望]
1、英文界面下整句提示（含括号内原因）均为英文。
2、同类错误走 i18n，避免硬编码中文。
```

对应命令示例（含项目名、版本、可选截图）：

```bash
node "skills/skill/mcp/scripts/zentao-bug-create.mjs" ^
  --project-name "位置监控平台-国际化" ^
  --title "【Message-Communication Logs】英文界面下发送失败提示括号内仍为中文" ^
  --steps-file "./tmp-bug-steps.md" ^
  --severity 3 --pri 3 ^
  --opened-build "管理后台-国际化V1.3" ^
  --attach "C:\path\to\screenshot.png"
```

### 排版与图片

- **异常空白**：旧版脚本把**每个空行**都转成 `<p> </p>`，禅道富文本会拉出大段间距；已改为合并空行并去掉空段落。若仍觉得紧，可减少 `steps-file` 里连续空行。
- **截图**：创建接口的 `steps` 字段仅为 **HTML 文本**，**不能**直接把本地图片二进制写入。请使用脚本参数 **`--attach`**（可多次）传入截图路径；创建成功后会调用 **`POST /api.php/v2/files`** 挂到该缺陷（需禅道 **v22+** 且服务端开放该接口）。若上传失败，可在禅道界面手动补传附件。

### Cursor 面板用户上传的图片如何进禅道（重要）

用户在 **Cursor 聊天里上传/粘贴的截图**，本质上会先**落盘为本机文件**（例如会话上下文中列出的 **`image_files`** 路径，或 `.cursor\projects\...\assets\`、工作区目录下的 `.png` 等）。禅道 REST **只接受「已有磁盘路径 → 附件接口」**，不能把聊天里的像素流直接「塞」进缺陷正文。

**AI 执行写入时必须做到：**

1、从当前对话中收集**所有**与本次缺陷相关的图片的**绝对路径**（以用户消息或工具可见的 `image_files`、工作区文件路径为准）。  
2、构造 `zentao-bug-create.mjs` 命令时，为每张图追加一个参数：**`--attach "完整路径"`**（Windows 路径含反斜杠时请用双引号包住整条路径；可重复多次 `--attach`）。  
3、路径优先使用对话里给出的路径；若用户只发了图未显示路径，可请用户将图片**另存到工作区某文件**（如 `bug-attach/1.png`）或**重新附加**，以便得到稳定路径后再执行脚本。

**示例（两张 Cursor 里上传的图）：**

```bash
node "skills/skill/mcp/scripts/zentao-bug-create.mjs" ^
  --project-name "位置监控平台-国际化" ^
  --title "【模块】问题简述" ^
  --steps-file "./tmp-bug-steps.md" ^
  --severity 3 --pri 3 ^
  --opened-build "管理后台-国际化V1.3" ^
  --attach "C:\Users\xxx\.cursor\projects\...\assets\image-abc123.png" ^
  --attach "C:\Users\xxx\Desktop\第一季度数据源\bug-attach\screen2.png"
```

**限制说明：** 若环境无法读取该文件路径（权限、文件已移动），附件会失败，需在禅道界面手动上传。图片**不会**自动插入「重现步骤」富文本正文中；若必须图文并茂，可在禅道里打开缺陷后编辑正文插入图片，或依赖附件区查看截图。

### 禅道「项目」与「产品」（重要）

- 接口层创建路径必须带**产品 ID**；脚本通过上表顺序**自动推断**，使用户侧只需关心 **「哪个项目」**（`--project-name` 关键词匹配项目名，与日常「进指定项目提缺陷」一致）。
- 若项目下**尚无任何缺陷**且项目详情里也没有产品字段，则无法推断，必须 **`--product-id`** 或先在项目中产生一条历史缺陷。

### 参数映射规则

| 用户提供的参数 | 对应脚本参数 | 默认值 | 说明 |
|---------------|-------------|--------|------|
| 项目名称 | `--project-name` | 无默认 | 匹配禅道项目；脚本自动解析/推断产品 ID 并写入「所属项目」 |
| 项目 ID | `--project-id` | 不填 | 已知数字时可单独使用（从该项目下缺陷推断产品）；或与 `--product-id` 联用 |
| 产品 ID | `--product-id` | 不填 | **备用**：自动推断失败时手动指定 |
| 测试模块/迭代 | `--execution` | 不填 | 指派到具体的测试执行/迭代 |
| 版本号 | `--opened-build` | trunk | 如 V2.5.8、V1.0.3 等 |
| 截图/附件 | `--attach` | 不填 | **可多次**；值为本机图片**绝对路径**（含 Cursor 聊天上传图落盘路径） |
| 指派成员 | 通过 API 补充设置 | 当前登录用户 | 创建后再调分配接口（见下方说明） |
| 严重程度 | `--severity` | 3 | 根据缺陷等级参考表自动判断 |
| 优先级 | `--pri` | 3 | 根据影响面判断 |

### 指派成员的额外处理

`zentao-bug-create.mjs` 本身不支持直接指派人。如果用户提供了指派成员需求，在 Bug 创建成功后，需通过禅道 API 额外调用分配接口：

```
PUT /api.php/v1/bugs/{bugId}/assignedTo?assignedTo={成员账号}
```

### 执行流程

1. 收集用户输入：项目名称 + 问题描述 + 可选参数（模块、版本、指派人）
2. 按 **Bug 标准格式模板** 输出完整报告，展示给用户确认
3. 用户确认后，将写入禅道用的正文整理为 **`steps-file`**（含 `[前置条件]`…四段），**有序条目一律改为 `1、` `2、` `3、`**，保存为 `{workspace}/tmp-bug-{timestamp}.md`
4. 若用户在 Cursor 中上传了截图：从对话中取得图片的**本地路径**，在命令中加入对应 **`--attach "路径"`**（每张图一项，可多项）
5. 构造 `zentao-bug-create.mjs` 命令并执行
6. 若需指派成员，追加调用分配接口
7. 向用户反馈结果：Bug ID + 禅道链接 + 分配状态（若已 `--attach`，提醒用户在禅道附件区核对）

### 注意事项

- 首次建议加 `--dry-run` 参数，只打印请求体不提交，让用户确认格式正确后再正式创建
- `--project-name` 的关键词需能匹配禅道**项目**全名中的子串（与 `zentao-bugs-summary.mjs --project-name` 规则一致）；不确定时可用禅道里的项目 ID：`--project-id`
- 新建项目、**零历史缺陷**时，自动推断可能失败，需 `--product-id` 或先在该项目下有一条缺陷数据
- 版本号格式需与禅道的版本命名保持一致
- 如果用户只说"发现了个Bug"但没提供详细信息，必须主动询问：
  1. 属于哪个项目？
  2. 在哪个模块/页面？
  3. 具体的现象是什么？
  4. 有没有截图或日志？

### 示例对话

**用户：**
> 星联应急叫应平台的 Communication Logs 页面，英文环境下提示文案有中文残留，帮我生成Bug并写入禅道

**AI 输出的 Bug 报告：**
> ## 【通信记录-会话窗口】英文环境下错误提示中英混排
>
> **前置条件**
>
> 1、界面语言为英文。
> 2、已进入 Message → Communication Logs，并可打开与终端的会话。
>
> **复现步骤**
>
> 1、在 Communication Logs 中打开与某终端的通信记录，进入含「Last Known Location」的会话窗口。
> 2、触发向终端发送失败且服务端返回「客服终端余额不足」类错误的场景。
> 3、查看会话区内系统提示气泡文案。
>
> **实际结果**
>
> 1、提示为英文前缀与中文原因混排，例如：Failed to send to terminal(客服终端余额不足，请充值后重试)。
> 2、括号内说明仍为中文，与同页英文 UI 不一致。
>
> **预期结果**
>
> 1、英文界面下，整句提示（含失败原因）应为完整英文，不应再出现未翻译的中文括号内容。
> 2、与同类错误提示的 i18n 资源一致（避免硬编码中文）。

**确认后执行的写入命令：**
> node skills/skill/mcp/scripts/zentao-bug-create.mjs --project-name "星联应急叫应平台" --title "【通信记录-会话窗口】英文环境下错误提示中英混排" --steps-file ./tmp-bug.md --severity 3 --pri 3 --opened-build V2.5.8
