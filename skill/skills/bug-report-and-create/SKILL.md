# Bug 报告输出 + 禅道自动写入

本技能实现两步工作流：先输出结构化 Bug 报告，再自动写入禅道。

## 第一步：输出结构化 Bug 报告

收到用户描述的问题后，按以下模板输出完整的 Bug 报告。输出前必须确认用户已提供足够的信息（项目名称、问题描述），若信息不足则主动询问。

### Bug 标准格式模板

```markdown
## 【{模块名}】{简短标题}

**前置条件**
1. {条件1}
2. {条件2}

**复现步骤**
1. {步骤1}
2. {步骤2}

**实际结果**
1. {实际现象1}，例如：{关键日志示例}
2. {实际现象2}

**预期结果**
1. {期望行为1}
2. {期望行为2}
```

### 输出规范

| 字段 | 规则 |
|------|------|
| 模块名 | 用方括号标注所属功能模块，如【求救群聊】【国际化-控制台】【套餐商城】 |
| 标题 | 15~30 字，包含模块 + 问题本质，如「会话内发送失败后错误提示中英混排」 |
| 前置条件 | 列出复现所需的前置状态，如语言环境、页面位置、账号权限等 |
| 复现步骤 | 每步一行，编号排列，操作路径清晰到可直接执行 |
| 实际结果 | 客观描述看到的现象，附关键日志原文（如有） |
| 预期结果 | 描述正确行为应是什么样的 |
| 序号（重要） | 凡需要有序排列的条目，一律使用 `1.` `2.` `3.`；禁止混用 `1、` `1)` `（1）` `-`。 |

### 缺陷等级参考（severity）

| 等级 | 含义 | 典型场景 |
|------|------|----------|
| 2 | 核心功能不可用、数据丢失风险 | 支付流程崩溃、消息发不出、登录失败 |
| 3 | 功能异常但不阻塞主流程 | UI错乱、提示文案错误、部分字段未翻译 |
| 4 | 轻微体验问题、建议优化 | 排版微调、文案不够优雅 |

### 优先级参考（pri）

| 值 | 含义 | 使用场景 |
|----|------|----------|
| 2 | 高 | 影响大量用户或核心链路 |
| 3 | 中 | 正常缺陷，常规处理 |
| 4 | 低 | 非紧急，可延后修复 |

## 第二步：自动写入禅道（直接写入，不创建文件）

用户确认 Bug 内容无误后，调用底层脚本将 Bug 直接写入禅道。

### 写入脚本

`skills/skill/mcp/scripts/zentao-bug-create.mjs`

### 构建命令

将四段正文直接拼接为 `--steps` 参数，不创建 `tmp-bug*.md` 等本地记录文件。

> 换行写法：在命令行里用字面量 `\n` 表示换行，脚本会自动解码为真实换行；序号行 `1.` / `1、` / `1)` 都会被脚本识别为有序列表项并渲染成禅道 `<ol>`，避免“密密麻麻挤成一段”。

```bash
node "skills/skill/mcp/scripts/zentao-bug-create.mjs" ^
  --project-name "{项目名称}" ^
  --title "{Bug 标题}" ^
  --steps "{[前置条件][步骤][结果][期望]拼接后的文本}" ^
  --severity {严重程度} ^
  --pri {优先级} ^
  --opened-build {版本号} ^
  --execution {迭代/测试模块ID}
```

### 写入正文结构（与第一步对齐）

第一步报告含 前置条件 / 复现步骤 / 实际结果 / 预期结果 四块时，写入 `--steps` 参数必须四块齐全。推荐格式：

```markdown
[前置条件]
1. ……
2. ……

[步骤]
1. ……
2. ……

[结果]
1. ……
2. ……

[期望]
1. ……
```

### 序号规范（写入 steps 时强制对齐）

1. 仅使用 `{n}.`（如 `1.` `2.` `3.`），每条独占一行。  
2. 每个小节（`[前置条件]`、`[步骤]`、`[结果]`、`[期望]`）都从 `1.` 重新编号。  
3. 若用户草稿里出现 `1、` / `（1）` / 多级编号，写入前统一改写为 `1.` `2.`。  
4. 小节标题和说明句独立成行，不加序号。

### 参数映射规则

| 用户提供的参数 | 对应脚本参数 | 默认值 | 说明 |
|---------------|-------------|--------|------|
| 项目名称 | `--project-name` | 无默认 | 匹配禅道项目 |
| 项目 ID | `--project-id` | 不填 | 已知数字时可单独使用 |
| 产品 ID | `--product-id` | 不填 | 自动推断失败时手动指定 |
| 测试模块/迭代 | `--execution` | 不填 | 指派到具体测试执行/迭代 |
| 版本号 | `--opened-build` | trunk | 如 V2.5.8、V1.0.3 等 |
| 指派成员 | 通过 API 补充设置 | 当前登录用户 | 创建后再调分配接口 |
| 严重程度 | `--severity` | 3 | 根据缺陷等级参考表自动判断 |
| 优先级 | `--pri` | 3 | 根据影响面判断 |

### 指派成员额外处理

`zentao-bug-create.mjs` 不支持直接指派人。若用户指定成员，在 Bug 创建成功后追加调用：

`PUT /api.php/v1/bugs/{bugId}/assignedTo?assignedTo={成员账号}`

### 执行流程

1. 收集用户输入：项目名称 + 问题描述 + 可选参数（模块、版本、指派人）。  
2. 按模板输出完整 Bug 报告，展示给用户确认。  
3. 用户确认后，将正文整理为 `--steps` 文本（含四段）。  
4. 构造并执行 `zentao-bug-create.mjs` 命令，直接写入禅道。  
5. 若需指派成员，追加调用分配接口。  
6. 反馈结果：Bug ID + 禅道链接 + 分配状态。

### 注意事项

1. 首次建议加 `--dry-run`，先只打印请求体供确认。  
2. `--project-name` 关键词需能匹配禅道项目全名子串，不确定时改用 `--project-id`。  
3. 新建项目且零历史缺陷时自动推断可能失败，需提供 `--product-id`。  
4. 版本号格式需与禅道版本命名一致。  
5. 如果用户只说“发现了个Bug”但信息不足，必须补问：  
   1. 属于哪个项目？  
   2. 在哪个模块/页面？  
   3. 具体现象是什么？  
   4. 有没有日志或可复现操作细节？

### 示例命令

```bash
node "skills/skill/mcp/scripts/zentao-bug-create.mjs" ^
  --project-name "【磐钴】位置监控平台-国际化项目" ^
  --title "【通信记录-会话窗口】英文界面下错误提示中英混排" ^
  --steps "[前置条件]\n1. 界面语言为英文。\n2. 已进入 Message → Communication Logs。\n\n[步骤]\n1. 打开会话窗口。\n2. 触发发送失败场景。\n3. 查看失败提示。\n\n[结果]\n1. 提示中出现中文残留。\n\n[期望]\n1. 英文界面下整句提示均为英文。" ^
  --severity 3 ^
  --pri 3 ^
  --opened-build "V2.5.8"
```
